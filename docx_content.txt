[19] [Heading 1] 1. The Problem — Why JavaScript Async Is Broken
[20] [Heading 2] 1.1 The Layered Mess of JavaScript's Async Evolution
[21] [Normal] JavaScript's asynchronous programming model has evolved through at least five distinct paradigms since 1995, each bolted onto the previous without ever replacing it. The result is a language where there is no single, coherent primitive for expressing asynchronous intent. Each iteration solved surface ergonomics while leaving the fundamental architectural flaws intact.
[22] [Normal] 
[23] [Normal] 
[24] [Heading 2] 1.2 The Four Fundamental Failures — Deep Analysis
[25] [Heading 3] 1.2.1 No True Cancellation
[26] [Normal] AbortController is a bolt-on solution that requires manual threading of a signal through every async call. It does not propagate automatically, has no connection to the call-stack lifetime, and does not compose with error handling. There is no built-in way to say "cancel this entire tree of operations if any sub-operation fails" — which is the defining property of structured concurrency.
[27] [Normal] The deeper issue: once asynchronous work is scheduled in JavaScript, there is no general mechanism to forcibly stop it. AbortController provides a signal that well-behaved code can observe — but it is a convention, not an enforcement mechanism. Promises, callbacks, and async functions represent results and continuations, not ownership of the underlying execution.
[28] [Normal] 
[29] [Normal] // AbortController: the bolt-on cancellation solution
[30] [Normal] async function fetchWithTimeout(url, timeoutMs) {
[31] [Normal]   const controller = new AbortController();
[32] [Normal]   const id = setTimeout(() => controller.abort(), timeoutMs);
[33] [Normal]   try {
[34] [Normal]     const response = await fetch(url, { signal: controller.signal });
[35] [Normal]     return await response.json();
[36] [Normal]   } finally {
[37] [Normal]     clearTimeout(id);  // must manually clean up
[38] [Normal]   }
[39] [Normal] }

[48] [Heading 3] 1.2.2 No Structured Concurrency
[49] [Normal] Structured concurrency is a programming paradigm pioneered in Kotlin (coroutines/CoroutineScope), Swift (task groups), Python 3.11 (TaskGroup), and Java 21 (Project Loom virtual threads). The core principle: the lifetime of concurrent tasks is lexically scoped — a task group cannot outlive the scope that created it, and if any task fails, all sibling tasks are automatically cancelled.
[50] [Normal] JavaScript has none of this. Promise.all(), Promise.allSettled(), and Promise.race() are utility combinators, not a concurrency model. The classic failure mode is the "ghost task" problem — abandoned async computations consuming CPU, memory, and network after the calling code has moved on.
[51] [Normal] 
[52] [Normal] // Classic unstructured concurrency: the ghost task problem
[53] [Normal] async function processUser(id) {
[54] [Normal]   const userFetch = fetch(`/api/user/${id}`);  // orphaned — no owner, never cancels
[55] [Normal]   throw new Error('early exit');              // fetch still runs, wastes resources
[56] [Normal]   const user = await userFetch;
[57] [Normal] }
[58] [Normal]  
[59] [Normal] // With Promise.all: one rejection does NOT cancel the others
[60] [Normal] async function loadDashboard(userId) {
[61] [Normal]   const [user, orders, analytics] = await Promise.all([
[62] [Normal]     getUser(userId),
[63] [Normal]     getOrders(userId),   // throws an error after 2 seconds
[64] [Normal]     getAnalytics(userId) // STILL RUNNING! Never cancelled! Resource leak!
[65] [Normal]   ]);
[66] [Normal] }

[68] [Heading 3] 1.2.3 Error Propagation Chaos — Three Incompatible Channels
[69] [Normal] JavaScript developers must navigate three distinct and incompatible error propagation mechanisms simultaneously. This fragmentation is not just aesthetic — it is a source of real bugs. A try/catch block around an async call that omits await silently fails to catch Promise rejections. Prof. Yizhou Zhang (Cornell/Waterloo) identified this as a critical safety gap: asynchronous exceptions are easily dropped since neither C# nor JavaScript statically checks exception types.
[70] [Normal] 
[71] [Normal] 
[72] [Heading 3] 1.2.4 No Resumable Control Flow — The Deepest Limitation
[73] [Normal] This is the most fundamental limitation — the one that makes true algebraic effects impossible to polyfill without engine changes. When JavaScript executes a throw statement, the engine walks the call stack upward, destroying each stack frame until it finds a matching catch clause. This process is irreversible.
[74] [Normal] Algebraic effects require the opposite behavior: perform suspends execution and preserves the entire suspended call stack as a delimited continuation. The handler receives the continuation as a first-class value and can choose to resume it, ignore it, or resume it multiple times. This requires engine-level support that does not exist in any JavaScript engine today.

[78] [Normal]   if (!cache.has(id)) {

[80] [Normal]     cache.set(id, user);                             // resume here after handler runs
[81] [Normal]   }
[82] [Normal]   return cache.get(id);
[83] [Normal] }

[90] [Heading 3] 1.2.5 The Function Coloring Problem (Nystrom 2015 — Extended Analysis)
[91] [Normal] In 2015, Bob Nystrom published "What Color is Your Function?" articulating a deep architectural problem: in languages like JavaScript, functions have a "color." Blue functions are synchronous. Red functions are async. The rules are asymmetric: async functions can call both types, but synchronous functions cannot call async functions in a value-extracting way. Every async function is "viral" — it infects every caller up the entire call stack.
[92] [Normal] This creates cascading refactors across entire codebases. If a previously synchronous operation (e.g., reading a config value) becomes asynchronous, every function that called it must be updated. This is not just ergonomic nuisance — it is an architectural constraint that bleeds async concerns into layers of code that should be implementation-agnostic.

[94] [Normal] // The function color problem in action
[95] [Normal] // Later, getConfig() becomes async. Now EVERY caller must become async:
[96] [Normal] async function getConfig() { /* was sync, now async */ }
[97] [Normal]  
[98] [Normal] // This function must now be async...
[99] [Normal] async function initApp() {
[100] [Normal]   const config = await getConfig();
[101] [Normal]   // ...and so must every function that calls initApp()...
[102] [Normal] }
[103] [Normal]

[112] [Heading 1] 2. Algebraic Effects — Theoretical Foundations
[113] [Heading 2] 2.1 Origins and Mathematical Basis
[114] [Normal] Algebraic effects were introduced by Gordon Plotkin and John Power in 2001–2003, and extended with effect handlers by Plotkin and Matija Pretnar in 2009–2013. The original work emerged from category theory and denotational semantics — a branch of theoretical computer science studying how programming language constructs can be given rigorous mathematical meaning.
[115] [Normal] The central insight: many computational side effects — reading/writing state, raising exceptions, performing I/O, generating random numbers, iterating — share a common algebraic structure. They can all be described as operations in an algebraic theory, and their semantics can be defined by giving handlers (interpretations) for those operations. This is directly analogous to how algebraic data types (like Option or Result) give a unified structure to data modeling.

[129] [Heading 1] 3. Language Survey — Who Solved This Best and How
[130] [Normal] Six languages have implemented algebraic effects natively or near-natively. Each provides critical lessons for a JavaScript library implementation. The architectural choices they made — particularly around type systems, compilation strategies, and continuation semantics — are directly applicable to TypeScript.
[131] [Heading 2] 3.1 Koka (Microsoft Research) — The Gold Standard
[132] [Normal] Koka is a strict functional language designed from scratch by Daan Leijen at Microsoft Research, with algebraic effects and row-polymorphic effect types at its core. It represents the most complete, production-oriented implementation of the theory and is the single most relevant reference point for a JavaScript library.
[133] [Normal] Key Design Features
[134] [List Paragraph] Row-polymorphic effect types: Every function's type signature explicitly tracks what effects it may perform — e.g., (int) -> <async, exn> int. No hidden side effects are possible.
[135] [List Paragraph] Evidence passing compilation: Koka avoids runtime stack manipulation by passing evidence (lightweight references to handlers) as implicit function arguments. This compiles to efficient C with no garbage collector pressure.
[136] [List Paragraph] Async/await as a library: Koka implements async/await, cancellation, and timeout as a user-space library on top of effect handlers — not as built-in syntax. This demonstrates the power of the primitive.
[137] [List Paragraph] Compiles to JavaScript: Koka's cooperative concurrency model maps naturally to the JS event loop — it already compiles to JavaScript for browser deployment.
[138] [List Paragraph] Performance: Koka often outperforms mature systems including C++ in algebraic-effect-heavy benchmarks, due to Perceus reference counting and in-place mutation optimization.

[153] [Heading 2] 3.2 OCaml 5 (Multicore OCaml) — Industrial-Strength Effects
[154] [Normal] OCaml 5.0 (released 2022) merged the Multicore OCaml project, introducing native algebraic effect handlers as the primary concurrency mechanism — the first mainstream systems language to do so. The retrofitting of effect handlers onto an existing language without breaking backward compatibility is directly analogous to the challenge of building a JS library.
[155] [Normal] Key Design Choices
[156] [List Paragraph] One-shot continuations only (by design): Enforces linearity at runtime, preventing double-resume bugs. This restriction improves performance dramatically.
[157] [List Paragraph] Shallow vs. deep handlers: OCaml supports both. Deep handlers automatically handle recursive continuations; shallow handlers handle just one step, enabling more control.
[158] [List Paragraph] Fibers as stack segments: Continuations are implemented as heap-allocated fiber stacks that grow on demand, supporting millions of concurrent lightweight threads.
[159] [List Paragraph] Eio library: Built on OCaml 5 effects, Eio provides structured concurrency (switches = scoped fiber lifetimes), direct-style IO, cancellation, and composable resource management — all in userland.
[160] [List Paragraph] Picos framework: A scheduler-agnostic interface for fibers, enabling structured and unstructured concurrency to coexist and interoperate without coupling to a specific runtime.
[161] [Normal] 
[162] [Heading 2] 3.3 Eff — The Research Prototype (Academic Ceiling)
[163] [Normal] Eff is a research language (Andrej Bauer & Matija Pretnar, University of Ljubljana) that served as the original experimental ground for algebraic effects and handlers. It supports multi-shot continuations and has strongly influenced all subsequent implementations. While not production-ready, Eff defines the theoretical ceiling of what algebraic effects can express.
[164] [List Paragraph] Multi-shot continuations: A handler can invoke the same continuation multiple times, enabling backtracking search, probabilistic branching, and save/load semantics.
[165] [List Paragraph] Effect inference: Eff infers effect types automatically — you do not need to annotate every function.
[166] [List Paragraph] Deep influence: Koka, OCaml effects, and nearly all academic papers trace back to Eff's semantics.
[167] [Heading 2] 3.4 Unison — Effects as First-Class Values
[168] [Normal] Unison, a content-addressed programming language, treats algebraic effects ("abilities") as a core language feature. Like Eff, it supports multi-shot continuations, allowing a handler to invoke a captured continuation multiple times — enabling co-operative multithreading, backtracking, and "forking the universe" computations. Unison's type system tracks abilities the same way Koka tracks effects.
[169] [Heading 2] 3.5 Effekt (Scala/JVM) — Effects Without Type-Level Madness
[170] [Normal] Effekt is a Scala library and research language (Brachthäuser, Schuster, Ostermann) that achieves effect safety and effect polymorphism through capability passing and intersection types — without complex type-level programming (no type classes, no HKTs). It implements multi-prompt delimited continuations via JVM bytecode transformation (CPS transformation at the bytecode level), enabling direct-style async without language changes.

[175] [Heading 1] 4. Existing JavaScript / TypeScript Libraries — Complete Survey (Updated April 2026)
[176] [Normal] As of April 2026, the JavaScript effects ecosystem has advanced significantly. Effect has released a major v4 beta, Effection has shipped v4, and new patterns have emerged. This section documents all major libraries with their current status, strengths, and critical gaps.
[177] [Heading 2] 4.1 Effect-TS (the "effect" package) — Updated: v4 Beta (February 2026)
[178] [Normal] Effect-TS is the most mature and comprehensive functional effects library in the TypeScript ecosystem. It has rapidly grown into a complete application framework. As of April 2026, Effect v4 is in beta — the most ambitious release yet, with a rewritten runtime, significantly smaller bundles, and a unified package ecosystem.
[179] [Heading 3] Effect v4 Beta — What Changed (February 2026)
[180] [List Paragraph] Rewritten fiber runtime: The core fiber runtime has been rewritten from scratch for lower memory overhead, faster execution, and simpler internals. Every Effect application benefits from these optimizations immediately.
[181] [List Paragraph] Dramatic bundle size reduction: A minimal program using Effect, Stream, and Schema drops from ~70 KB (v3) to ~20 KB (v4). The core effect package with aggressive tree-shaking bundles to ~6.3 KB minified + gzipped.
[182] [List Paragraph] Unified versioning: All ecosystem packages now share a single version number and are released together. In v3, packages were versioned independently, making compatibility difficult to track.
[183] [List Paragraph] Unstable modules (effect/unstable/*): 17 unstable modules now ship inside the core effect package: AI, CLI, cluster, devtools, eventlog, HTTP, httpapi, jsonschema, observability, persistence, process, reactivity, RPC, schema, SQL, workflow, workers.
[184] [List Paragraph] @effect/ai SDK: Released as an alpha alongside v4. The Effect AI SDK brings algebraic effect patterns to AI agent orchestration — structured concurrency for LLM calls, typed error handling for model failures, fiber-based parallelism for multi-agent systems.

[207] [Heading 3] Critical Gaps (Still Present in v4)
[208] [List Paragraph] Monadic encoding, not true algebraic effects: Effect-TS v4 is still built on monads (the IO monad pattern from Haskell), not on resumable delimited continuations. You cannot "resume from where you performed an effect" — you describe workflows as data structures that are interpreted later.
[209] [List Paragraph] No true multi-shot continuations: Effect-TS fibers are one-way — once you fork or schedule, there is no handler that can resume the same computation from a different point.
[210] [List Paragraph] Function color problem remains: Code written in Effect style is not interoperable with ordinary Promise-based code without explicit bridging. The "async plague" is replaced with the "Effect plague."
[211] [List Paragraph] Steep learning curve: Even with v4's improvements, Effect requires a paradigm shift to functional, value-oriented programming. Teams without FP background find initial adoption slow.
[212] [List Paragraph] Restricted environment edge cases: v4 beta introduced a keepAlive mechanism (globalThis.setInterval) that breaks in restricted runtimes like Convex — resolved in later beta versions, but illustrates the challenge of portability.
[213] [Normal] 
[214] [Heading 2] 4.2 Effection — v4 Released (December 2025)
[215] [Normal] Effection (from the Frontside team) takes a different philosophy from Effect-TS: rather than introducing a new paradigm, it extends JavaScript's existing concurrency primitives with structured concurrency semantics while feeling as native as possible. Effection v4 was released in December 2025, addressing v3's key limitations.
[216] [Heading 3] Effection v4 — What Changed (December 2025)
[217] [List Paragraph] Parent-task priority: In v4, a parent task always has priority over its children. This makes task execution more predictable and allows parents to supervise children properly. This changes execution ordering from v3.
[218] [List Paragraph] Explicit scope management: v4 introduces explicit scope boundaries. In v3, new scopes were created as a by-product of action()/call()/spawn() — confusing users. In v4, scope boundaries are first-class and visible.
[219] [List Paragraph] Native JavaScript stack traces: Effection v4 programs "live and breathe on the JavaScript stack" — stack traces are native, readable by IDEs, LSPs, and debuggers. This was a major complaint about v3 and most effects libraries.
[220] [List Paragraph] EffectionX Extensions: Ready-to-use libraries for common patterns: WebSockets with automatic cleanup, process management, worker threads with structured concurrency. Work with both v3 and v4.
[221] [List Paragraph] Back-ported to v3.2.0+: The native stack trace feature was back-ported to Effection v3.2.0 so existing apps can benefit without migrating.

[252] [Heading 1] 5. Academic Research Landscape — Complete Bibliography with Annotations
[253] [Normal] The academic literature on algebraic effects is rich and directly applicable to a JavaScript implementation. This section catalogues all relevant papers with annotations explaining their practical relevance to the library you are building.
[254] [Heading 2] 5.1 Foundational Papers (Mandatory Reading)
[255] [Heading 3] Plotkin & Power — Algebraic Operations and Generic Effects (2003)
[256] [Normal] The mathematical origin of the entire field. Establishes the categorical semantics of algebraic effects — that computational effects can be modelled as operations of an algebraic theory, and that programs are free models of these theories. Every subsequent paper builds on this foundation. Essential for understanding why effects compose when monads do not.
[257] [Heading 3] Plotkin & Pretnar — Handling Algebraic Effects (2009/2013)
[258] [Normal] Introduces effect handlers as the operational mechanism — the catch-like construct that intercepts effects and receives the delimited continuation. This paper defines the core semantics your library will implement. The 2009 version appeared at ESOP; the expanded 2013 version appeared in Logical Methods in Computer Science. Both are essential reading.
[259] [Heading 3] Bauer & Pretnar — Programming with Algebraic Effects and Handlers (2015)
[260] [Normal] The operational introduction, using the Eff language. Concrete programming examples showing how exceptions, generators, async, state, and non-determinism are all special cases of algebraic effects with handlers. The most accessible entry point to the theory for working programmers.
[261] [Heading 2] 5.2 Implementation & Compilation Papers
[262] [Heading 3] Leijen — Algebraic Effects for Functional Programming (MSR-TR-2016-29)
[263] [Normal] Daan Leijen's Microsoft Research technical report presenting Koka's implementation strategy — evidence passing, row-polymorphic types, and efficient compilation to C. Directly applicable to compiling to JavaScript. The evidence-passing strategy is the most important insight for building a zero-overhead TypeScript library. Available at microsoft.com/en-us/research.
[264] [Heading 3] Leijen — Type Directed Compilation of Row-Typed Algebraic Effects (POPL 2017)
[265] [Normal] Formal treatment of Koka's type system. Shows how effect rows (tracking which effects a function may perform) can be compiled to efficient evidence-passing code without runtime overhead. The theoretical foundation for TypeScript effect row encoding via intersection types.
[266] [Heading 3] Sivaramakrishnan et al. — Retrofitting Effect Handlers onto OCaml (PLDI 2021)
[267] [Normal] The definitive paper on how OCaml 5 was engineered. Covers the implementation of fibers as heap-allocated stack segments, design of one-shot continuations, DWARF compatibility, and benchmarks against monadic libraries. arXiv:2104.00250. Critical reading for understanding the tradeoffs between one-shot and multi-shot implementations.
[268] [Heading 3] Dolan et al. — Concurrent System Programming with Effect Handlers (2018)
[269] [Normal] Demonstrates that effect handlers can express difficult concurrent programs (web servers, co-operative multithreading) at performance parity with optimized monadic libraries. Validates the practical viability of effects for systems programming.
[270] [Heading 3] Brachthäuser, Schuster, Ostermann — Effect Handlers for the Masses (OOPSLA 2018)
[271] [Normal] Introduces the Effekt language and its capability-passing approach. Achieves effect polymorphism and safety without type-level programming, making the model accessible to mainstream developers. This ergonomics insight should directly inform your API design — capability passing is how you get type safety without complex row-polymorphic types in TypeScript.
[272] [Heading 2] 5.3 Directly Relevant to JavaScript Implementation
[273] [Heading 3] Kawahara & Kameyama — One-Shot Algebraic Effects as Coroutines (TFP 2020)
[274] [Normal] Proves that one-shot algebraic effects are macro-expressible as asymmetric coroutines (and vice versa is not true). Since JavaScript generators are asymmetric coroutines, this paper provides the theoretical foundation and proof of correctness for implementing one-shot effects via generators. Essential reading before writing the first line of implementation code.
[275] [Heading 3] Cong & Asai — Expressive Power of One-Shot Control Operators and Coroutines (APLAS 2023)
[276] [Normal] Extends the Kawahara & Kameyama result. Provides a rigorous mathematical comparison of one-shot effect handlers, delimited continuations, and coroutines. Confirms that generators (asymmetric coroutines) are strictly less expressive than multi-shot effect handlers, and that one-shot effects and coroutines are equivalent. Key theoretical grounding for your Phase 1 (generator-based) vs Phase 3 (multi-shot) design decision.
[277] [Heading 3] Ahman & Pretnar — Asynchronous Effects (LMCS 2024, arXiv:2307.13795)
[278] [Normal] Directly relevant to the JavaScript problem. Explores asynchronous algebraic effects — decoupling the signalling of an effect from its resolution, enabling non-blocking operation calls. Includes a prototype in OCaml and Agda type-safety proofs. This is the theoretical model for your "async effects" feature: a perform call that does not block the calling fiber while the effect is being resolved.
[279] [Heading 3] Zhang, Salvaneschi & Myers — Handling Bidirectional Control Flow (OOPSLA 2020, Cornell)
[280] [Normal] Introduces bidirectional algebraic effects — extending the model so that generators (async iterators) and async/await can interact bidirectionally. Proves that all declared effects are handled and none are accidentally handled. Available at cs.cornell.edu/andru/papers/ufo/bidirectional-effects.pdf. Critical for the Phase 3 async iterator integration.
[281] [Heading 2] 5.4 Newest Research (2024–2026) — Not in Original Document
[282] [Heading 3] Ma & Zhang — Lexical Effect Handlers: Fast by Design, Correct by Proof (OOPSLA 2025)
[283] [Normal] Published at SPLASH/OOPSLA 2025. Introduces Lexical Effect Handlers — a restriction of algebraic effect handlers that enables significant performance improvements (equivalent to direct-style programming) while maintaining correctness guarantees. The "lexical" restriction means handlers are lexically visible, enabling ahead-of-time optimization. Directly applicable to a TypeScript implementation that wants zero-overhead hot paths.
[284] [Heading 3] Kammar, McCusker, Phipps-Costin & Lindley — Equational Axiomatization of Dynamic Threads (POPL 2026)
[285] [Normal] Published at POPL 2026. Provides an equational axiomatization of dynamic threads via algebraic effects, using presheaves on finite relations. Establishes the formal semantic foundations for concurrent algebraic effects — directly relevant to structured concurrency semantics in your library. arXiv citation: 2026 POPL proceedings.
[286] [Heading 3] Phipps-Costin et al. — Continuing WebAssembly with Effect Handlers (OOPSLA 2023 — Implementation Advancing)
[287] [Normal] Introduces WasmFX — effect handlers for WebAssembly. As of 2025, WasmFX is being standardized in the WebAssembly Stack Switching subgroup. An implementation exists in wasmtime and wasm-tools. V8 and SpiderMonkey implementations are being tracked. This is the platform-level primitive your library may eventually be able to exploit for true native performance.
[288] [Heading 2] 5.5 WasmFX & Stack Switching — The Platform Horizon
[289] [Normal] WasmFX extends WebAssembly with effect handlers as a unifying mechanism to enable efficient compilation of control idioms: async/await, generators/iterators, first-class continuations, lightweight threads. The design adds only three main instructions: cont.new (create a continuation), resume (run a continuation under a handler), and suspend (suspend with a tag to the nearest handler).
[290] [List Paragraph] Standardization status (2025): WasmFX is being considered as the basis for "stack switching" in WebAssembly. The proposal is known as "typed continuations." Active discussion in the WebAssembly Stack Switching subgroup.
[291] [List Paragraph] Implementations: Wasmtime (Rust-based Wasm runtime) and wasm-tools both have WasmFX implementations. Browser implementations are not yet available.
[292] [List Paragraph] Strategic implication for your library: Once WasmFX is standardized and browser-supported, a JavaScript library built on WasmFX continuations could achieve true native algebraic effects semantics — not via generator polyfills, but via actual platform-level delimited continuations. Build the library now; design the API with WasmFX as the eventual backend in mind.

[296] [Heading 1] 6. Implementation Strategy for Your Library
[297] [Normal] This section provides a detailed, technically grounded implementation roadmap based on the academic literature, lessons from existing libraries, and the unique constraints of the JavaScript/TypeScript ecosystem.
[298] [Heading 2] 6.1 Core Design Decisions
[299] [Heading 3] Decision 1: Generator-Based Delimited Continuations (Phase 1)
[300] [Normal] JavaScript generators (function*) are the natural implementation substrate for one-shot delimited continuations. A suspended generator IS a delimited continuation — it represents the "rest of the computation" from the yield point. The theoretical basis is Kawahara & Kameyama (TFP 2020): one-shot effects and asymmetric coroutines are equivalent in expressiveness.

[302] [Normal] // Core implementation concept (simplified)
[303] [Normal] function withHandler<T>(handler: Handler, thunk: () => Generator): T {
[304] [Normal]   const gen = thunk();
[305] [Normal]   let result = gen.next();
[306] [Normal]   while (!result.done) {
[307] [Normal]     const effect = result.value;           // the 'perform' call
[308] [Normal]     const continuation = (v: any) => gen.next(v);  // delimited continuation
[309] [Normal]     const resume = handler.handle(effect, continuation);
[310] [Normal]     result = gen.next(resume);
[311] [Normal]   }
[312] [Normal]   return result.value;
[313] [Normal] }

[348] [Heading 3] Decision 3: Structured Concurrency as an Algebraic Effect
[349] [Normal] Spawn and cancel are themselves effects, not primitives. This is the key insight from Effection and from the OCaml Eio library. A Spawn effect is performed to start a child task; a Cancel effect is performed to cancel a task. The structured concurrency guarantees come from the handler, not from the runtime — making the entire concurrency model testable and composable.
[350] [Normal] 
[351] [Normal] // Structured concurrency as algebraic effects
[352] [Normal] type SpawnEffect<T> = { kind: 'spawn', task: () => Effect<T, any> }
[353] [Normal] type CancelEffect   = { kind: 'cancel', taskId: TaskId }
[354] [Normal] type JoinEffect<T>  = { kind: 'join',   tasks: TaskId[] }
[355] [Normal]  
[356] [Normal] function* fetchBoth(urlA: string, urlB: string): Effect<[Response, Response], SpawnEffect<any> | CancelEffect> {
[357] [Normal]   const taskA = yield { kind: 'spawn', task: () => fetchUrl(urlA) };
[358] [Normal]   const taskB = yield { kind: 'spawn', task: () => fetchUrl(urlB) };
[359] [Normal]   try {
[360] [Normal]     const [a, b] = yield { kind: 'join', tasks: [taskA, taskB] };
[361] [Normal]     return [a, b];
[362] [Normal]   } catch (e) {
[363] [Normal]     yield { kind: 'cancel', taskId: taskA };  // guaranteed cleanup
[364] [Normal]     yield { kind: 'cancel', taskId: taskB };  // guaranteed cleanup
[365] [Normal]     throw e;
[366] [Normal]   }
[367] [Normal] }

[371] [Heading 2] 6.3 Critical Gaps to Fill Over Existing Work
[372] [List Paragraph] Multi-shot continuations via generator cloning: effects.js has a prototype. The gap is a production-quality, typed, GC-safe implementation. The Eff language semantics paper (Bauer & Pretnar 2015) provides the reference implementation.

[377] [List Paragraph] Zero-overhead testing: The killer feature missing from all existing libraries: a seamless "run with mock handlers" story that requires ZERO changes to the code under test. Not even a test helper import. Just swap the handler.
[378] [List Paragraph] WasmFX backend (Phase 4+): Design the API now so that the generator-based implementation can be transparently replaced with a WasmFX backend when browsers support it. The API surface should not change.

[381] [Heading 1] 7. New Frontier: Algebraic Effects for AI Agent Orchestration (2026)
[382] [Normal] A new and rapidly growing use case has emerged in 2025–2026: algebraic effects as the natural primitive for AI agent orchestration. This section explores why effects are uniquely well-suited to AI agent systems and how the ecosystem is already converging on this pattern.
[383] [Heading 2] 7.1 Why AI Agents Need Algebraic Effects
[384] [Normal] Modern AI agent systems face exactly the same structural problems as JavaScript's async model — amplified by orders of magnitude:
[385] [List Paragraph] Non-deterministic execution: LLM calls may succeed, fail, hallucinate, or timeout. Error propagation must be typed and composable.
[386] [List Paragraph] Parallel tool calls: Agents routinely call multiple tools in parallel. Without structured concurrency, failed tool calls leave ghost tasks consuming API rate limits.
[387] [List Paragraph] Retry and backoff: Every LLM provider requires retry logic. This is a natural algebraic effect — the business logic just "performs a Generate effect"; the handler decides retry policy.
[388] [List Paragraph] Dependency injection for models: Tests should run against a mock LLM, production against a real one. Effect handlers make this trivial — same code, different handler.
[389] [List Paragraph] Stateful multi-step workflows: Agent workflows with memory, tool use, and reflection are complex state machines. Algebraic effects are the natural primitive for expressing this.
[390] [List Paragraph] Observability: Every LLM call is an "effect." Wrapping effects with logging, tracing, and cost tracking is trivial — just compose the handler.

[430] [Heading 1] 8. TC39 Standardization Status & Platform Developments (April 2026)
[431] [Heading 2] 8.1 TC39 — No Proposal for Algebraic Effects
[432] [Normal] As of April 2026, there is no formal TC39 proposal for algebraic effects in JavaScript at any stage of the standardization process. The ECMAScript 2025 (16th edition) introduced Array.fromAsync, Error.isError, and explicit resource management with using. The February 2026 TC39 meeting advanced Float16Array, RegExp.escape, and Duplicate Named Capture Groups to Stage 4. None address the fundamental async control flow limitations.
[433] [Normal] The barriers to a TC39 proposal for algebraic effects remain substantial:
[434] [List Paragraph] Engine implementation complexity: Delimited continuations require fundamental changes to how V8, SpiderMonkey, and JavaScriptCore manage call stacks. These engines would all need significant modifications.
[435] [List Paragraph] Backward compatibility: Any syntax for perform or handle must not conflict with existing valid JavaScript identifiers and patterns. The esdiscuss.org archives from 2016 show Sebastian Markbåge first raising this with TC39 — it generated discussion but no action.
[436] [List Paragraph] Multi-shot semantics with SharedArrayBuffer: Multi-shot continuations have unclear semantics in the presence of shared mutable state — a show-stopper for browsers with SharedArrayBuffer enabled.
[437] [List Paragraph] Performance concerns: Capturing and preserving continuations is more expensive than stack unwinding. The performance impact on the vast majority of JavaScript code that does not use effects must be demonstrably minimal.
[438] [Normal] 
[439] [Heading 2] 8.2 What TC39 IS Working On (Adjacent Developments)
[440] [List Paragraph] Explicit resource management (using / await using): Now at Stage 4. Provides deterministic cleanup for resources — a partial solution to the structured lifetime problem. Not resumable, but solves resource leak issues.
[441] [List Paragraph] TypeScript compiler rewrite in Go: Microsoft is rewriting tsc and tsserver in TypeScript 6.0 will be the last TypeScript-based version. Future versions in Go should offer 10x performance improvement. This opens the door for more sophisticated type-level effect tracking without performance penalty.
[442] [List Paragraph] Node.js native TypeScript (since Node.js 22.18.0, July 2025): Node.js now natively supports TypeScript — no build step needed. This reduces the barrier to adopting TypeScript-first libraries like yours.
[443] [Normal] 
[444] [Heading 2] 8.3 WasmFX as the Platform Path
[445] [Normal] The most credible path to native algebraic effects in the browser runs through WebAssembly, not TC39. WasmFX (the "stack switching" / "typed continuations" proposal) provides delimited continuations at the WebAssembly level. If this proposal is standardized and implemented in V8 and SpiderMonkey, it would become possible to expose delimited continuations to JavaScript via a thin API — at near-native performance.
