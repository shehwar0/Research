2. WHAT ARE ALGEBRAIC EFFECTS?
================================================================================

Algebraic effects are a programming paradigm that separates WHAT a program
needs (effects) from HOW those needs are fulfilled (handlers). Think of them
as a more powerful, composable, and type-safe alternative to:

  - try/catch (error handling)
  - async/await (asynchronous programming)
  - dependency injection (service location)
  - middleware (request processing)

The key idea: A function "performs" an effect (like requesting data), and a
"handler" somewhere up the call stack decides what to do. The function is
then "resumed" with the result. This is like a super-powered yield/resume.

Example:

  // WHAT the program needs (effect declaration)
  function* fetchUser(id) {
    return yield* perform({ _tag: 'Fetch', url: `/users/${id}` })
  }

  // HOW it's fulfilled (handler)
  const result = run(function* () {
    return yield* handle(httpHandler, function* () {
      const user = yield* fetchUser(1)
      return user.name
    })
  })

The magic: You can swap the handler without changing the code. For testing,
use a mock handler. For production, use a real HTTP handler. For caching,
wrap with a caching handler. All without touching fetchUser().

3. HOW THE LIBRARY WORKS
================================================================================

CORE MECHANISM: Generator-Based Delimited Continuations

JavaScript generators (function*) are used as asymmetric coroutines.
When a function "performs" an effect, it yields a signal. The handler
loop intercepts this signal, processes it, and resumes the generator
with the result.

  function* myComputation() {
    const x = yield* perform({ _tag: 'Read', key: 'name' })
    //        ^^^^^^ This yields the effect signal up to the handler
    //               The handler processes it and sends a value back
    return `Hello, ${x}!`
  }

The handler loop (simplified):

  function handle(handler, computation) {
    const gen = computation()
    let input = undefined

    while (true) {
      const step = gen.next(input)     // Drive the generator
      if (step.done) return step.value // Computation finished

      const effect = step.value        // An effect was performed
      handler.handle(effect, (value) => {
        input = value                  // Handler provides a value
      })
    }
  }

This is based on the theoretical proof by Kawahara & Kameyama (2020) that
one-shot delimited continuations are equivalent to asymmetric coroutines.

TYPE SAFETY:

The library uses TypeScript's type system to track which effects a
computation performs:

  type Eff<ReturnType, Effects> = Generator<Effects, ReturnType, any>

  // This function performs FetchEffect and LogEffect
  function* myFn(): Eff<string, FetchEffect | LogEffect> { ... }

The compiler ensures all effects are handled before execution. If you
forget to handle an effect, you get a compile-time error.

KEY FEATURES:

  a) perform / handle — The core effect mechanism
     - perform() yields an effect signal from a generator
     - handle() intercepts signals and provides values via resume
     - One-shot resume guarantee (resume can only be called once)

  b) Handler Composition
     - compose() merges two handlers into one
     - composeAll() merges an array of handlers
     - Handlers can be nested (inner handlers shadow outer ones)

  c) Structured Concurrency
     - Scope: parent-child lifetime management
     - spawn: create child tasks within a scope
     - all: run tasks concurrently, wait for all
     - race: run tasks concurrently, take first result
     - Cancellation propagates from parent to children

  d) Typed Error Handling
     - fail() performs a typed error effect
     - catchFail() catches it and returns Result<T, E>
     - recover() provides a fallback value
     - retry() re-executes on failure with configurable attempts

  e) Promise Interop
     - fromPromise() wraps a Promise as an effect
     - toPromise() converts an effectful computation to a Promise
     - Bridges the gap between effects and existing async code

  f) AI Agent Effects
     - generate() — LLM text generation effect
     - useTool() — tool invocation effect
     - remember()/recall() — memory effects
     - observe() — observability/logging effect
     - Mock handlers for zero-change testing

5. PHASE 2 — STRUCTURED CONCURRENCY & ADVANCED HANDLERS (24 Tests)
================================================================================

Phase 2 added production-grade concurrency and performance optimizations.

FILES CREATED:
  - src/resource.ts  — Scoped resource lifecycle management
  - src/taskgroup.ts — Task groups with supervision strategies
  - src/evidence.ts  — O(1) evidence-passing handler dispatch
  - src/infer.ts     — Type-level effect row inference utilities

KEY FEATURES:

  a) Resource Management (resource.ts)
     Based on: OCaml Eio Switch pattern

     - Resource<T>: value + release function
     - using(): acquire a resource via effect, auto-released on scope close
     - bracket(): acquire/use/release combinator (like try-with-resources)
     - ResourceScope: tracks resources, LIFO release order
     - AggregateResourceError: collects all cleanup errors
     - Symbol.dispose integration (TC39 Explicit Resource Management)

     Example:
       yield* bracket(
         () => openFile('data.txt'),      // acquire
         function* (file) { return read(file) }, // use
         (file) => file.close(),          // release (guaranteed)
       )

  b) Task Groups (taskgroup.ts)
     Based on: Python trio Nursery, Kotlin CoroutineScope

     Three supervision strategies:
     - failFast: Cancel all siblings on first failure (strictest)
     - allSettled: Wait for ALL tasks, collect all results/errors
     - bestEffort: Cancel remaining on failure, don't re-throw

     Example:
       const results = await parallel([
         function* () { return yield* fetchUser(1) },
         function* () { return yield* fetchUser(2) },
       ], 'failFast')

  c) Evidence-Passing Handler Dispatch (evidence.ts)
     Based on: Leijen, POPL 2017 (Koka language)

     Problem: Standard handler dispatch walks the handler stack to find
     the right handler. This is O(n) where n = handler depth.

     Solution: Maintain an "evidence vector" that maps effect tags to
     handlers. Lookup is O(1) — just a Map.get().

     - EvidenceVector: hierarchical map with parent chaining
     - withEvidence(): install a handler in the evidence vector
     - runWithEvidence(): top-level runner with O(1) dispatch

  d) Type Inference Helpers (infer.ts)
     Compile-time type utilities:
     - InferEffects<T>: extract the effect row from a computation type
     - RequiresHandling<T>: check if unhandled effects remain
     - HandlerFor<E>: derive the handler type from an effect type
     - PipeHandled<A, Row, Handlers>: track effect elimination
     - ContainsEffect, ExtractEffect, EffectTagsOf, etc.

6. PHASE 3 — ADVANCED FEATURES (34 Tests)
================================================================================

Phase 3 implemented research-level features from recent academic papers.

FILES CREATED:
  - src/multishot.ts      — Multi-shot continuations
  - src/async-effects.ts  — Async effects (signal/execute/interrupt)
  - src/bidirectional.ts  — Bidirectional effect streams
  - src/lexical.ts        — Zero-overhead lexical handlers

KEY FEATURES:

  a) Multi-Shot Continuations (multishot.ts)
     Based on: Cong & Asai, APLAS 2023

     Problem: JavaScript generators are one-shot — once consumed, they
     can't be replayed. But multi-shot continuations are needed for
     backtracking, non-determinism, and probabilistic programming.

     Solution: Replay-based cloning. Record every value sent to gen.next()
     during execution. To "fork," create a fresh generator and replay
     the recorded input sequence.

     Built-in effects:
     - choose([1,2,3]): non-deterministic choice
     - amb(): boolean ambiguity (coin flip)
     - collectAll(): enumerate ALL possible results

     Example:
       const results = collectAll(function* () {
         const x = yield* choose([1, 2])
         const y = yield* choose([10, 20])
         return x + y
       })
       // results === [11, 21, 12, 22]

  b) Async Effects (async-effects.ts)
     Based on: Ahman & Pretnar, LMCS 2024

     Traditional effects BLOCK at the perform site. Async effects
     DECOUPLE signaling from execution:

     1. SIGNAL — Start the operation, get a handle immediately
     2. EXECUTE — Handler runs the operation in the background
     3. INTERRUPT — Resume the computation when the result is ready

     This enables pipeline parallelism without explicit fibers:

     Example:
       const result = await runAsyncEffects(function* () {
         const h1 = yield* signalAsync({ execute: () => fetch('/a') })
         const h2 = yield* signalAsync({ execute: () => fetch('/b') })
         // Both fetches running in parallel!
         return [yield* awaitHandle(h1), yield* awaitHandle(h2)]
       })

  c) Bidirectional Effects (bidirectional.ts)
     Based on: Zhang et al., OOPSLA 2020

     Standard effects are unidirectional: perform down, resume up.
     Bidirectional effects add a second channel: the computation can
     YIELD values upstream while PERFORMING effects downstream.

     This naturally models streaming producers that need resources:

     - yieldValue(): yield a value upstream
     - collectStream(): collect all yielded values
     - mapStream/filterStream/takeStream: stream combinators
     - toAsyncIterable(): bridge to for-await-of syntax

     Example:
       function* readLines(path): EffectStream<string, void, ReadEffect> {
         const content = yield* perform({ _tag: 'Read', path })
         for (const line of content.split('\n')) {
           yield* yieldValue(line)  // Send each line upstream
         }
       }

  d) Lexical Handler Optimization (lexical.ts)
     Based on: Ma & Zhang, OOPSLA 2025

     Problem: Every effect perform has overhead from:
       1. Generator yield (context switch)
       2. Handler dispatch (finding the handler)
       3. Resume callback creation (closure allocation)
       4. Generator resume (gen.next())

     Solution: For "tail-resumptive" handlers (where resume is the last
     call), eliminate ALL overhead. The handler becomes a direct function
     call — no yield, no resume callback, no closure.

     - createLexicalHandler(): build zero-overhead handler
     - runLexical(): drive computation with zero overhead
     - isTailResumptive(): detect promotable handlers
     - toLexicalHandler(): promote standard handler to lexical
     - benchmarkHandler(): compare performance

     - Deferred settlement for task groups (microtask ordering)

KNOWN LIMITATIONS:

  - One-shot by default: Generators are consumed on use. Multi-shot
    requires replay, which has O(n) cost per fork.

  - No true parallelism: JavaScript is single-threaded. Structured
    concurrency is cooperative, not preemptive.

  - WasmFX not yet available: The WasmFX backend is a stub. True
    native stack switching requires browser support (in progress).

  - Async effects need explicit runner: runAsyncEffects() must be
    used for the signal/await pattern (cannot use standard handle()).

================================================================================

  a) PERFORMANCE
     - The generator yield/resume has inherent overhead (~2-5x vs
       direct function calls). When WasmFX ships, this drops to ~0.
     - The lexical handler optimization already eliminates most
       overhead for tail-resumptive handlers.
     - Consider adding a JIT-compiled fast path for hot effect loops.

  b) DEVELOPER EXPERIENCE
     - Add a TypeScript compiler plugin for automatic effect inference
       (currently manual type annotations are needed).
     - Build a VS Code extension for effect visualization.
     - Add an interactive playground (web-based REPL).

  c) ECOSYSTEM
     - Framework adapters: Next.js, Hono, tRPC, Express.
     - React integration (useEffect replacement with algebraic effects).
     - Database driver effects (Prisma, Drizzle integration).

  d) MULTI-SHOT PERFORMANCE
     - Current replay-based cloning is O(n) per fork. Could be
